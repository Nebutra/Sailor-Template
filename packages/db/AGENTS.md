# AGENTS.md — packages/db

Execution contract for Nebutra's database package.

## Scope

Applies to everything under `packages/db/`.

## What This Package Owns

`@nebutra/db` owns the Prisma schema, migration history, generated Prisma
client, and the shared database runtime used across the monorepo.

## Source Of Truth

- Database schema: `prisma/schema.prisma`
- Prisma runtime config: `prisma.config.ts` and `prisma/prisma.config.ts`
- Migration history: `prisma/migrations/`
- Seed entrypoints: `prisma/seed.ts` and `scripts/seed-model-configs.ts`
- Public package surface: `src/index.ts`
- Shared client and tenancy boundaries: `src/client.ts`, `src/rls.ts`, `src/pool.ts`

If behavior is derived from the schema, change the schema first and regenerate
the client in the same change.

## Defaults

- Prefer `getTenantDb()` for request-scoped application code. Treat
  `getSystemDb()` as an explicit escape hatch.
- Preserve the RLS contract around `app.current_org_id` when changing tenant
  access code or migration SQL.
- Keep package exports intentional. If runtime access patterns change, update
  `src/index.ts` and consuming types together.
- Do not use `db:push` as the default path for durable schema changes. Prefer
  checked-in migrations.

## Schema And Migration Rules

- Start schema changes in `prisma/schema.prisma`.
- For durable schema changes, create or update a migration under
  `prisma/migrations/` and keep the SQL aligned with the schema change.
- Do not silently rewrite existing shipped migrations unless the task is
  explicitly about repairing migration history.
- When changing tenant-scoped tables or policies, review the RLS implications as
  part of the same change.
- When removing or renaming models, enums, or columns that are part of the
  public package surface, update exports and downstream usage in the same work.

## Generated And Derived Files

- `src/generated/prisma/` is generated by Prisma. Do not hand-edit it.
- Regenerate the client after schema or generator changes with
  `pnpm --filter @nebutra/db db:generate`.
- Treat local database state, Prisma Studio output, and ad hoc generated files
  as derived artifacts, not source.

## Validation

Run the smallest credible validation for the change:

```bash
pnpm --filter @nebutra/db typecheck
```

If the schema, generator, or exported Prisma types changed, also run:

```bash
pnpm --filter @nebutra/db db:generate
pnpm --filter @nebutra/db typecheck
```

If migrations or seeds changed, validate them against a local dev database with
the narrowest relevant command, usually:

```bash
pnpm --filter @nebutra/db db:migrate
pnpm --filter @nebutra/db db:seed
```
