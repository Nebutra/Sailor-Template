# Figma Onboarding — Tokens Studio

A 5-minute setup so a designer can pull Nebutra design tokens into Figma and start syncing changes back to git.

> See [`figma-sync.md`](./figma-sync.md) for the full architecture and CI behaviour.

---

## 1. Install the plugin

Tokens Studio for Figma is a **Figma plugin**, not an npm package. Install it from the Figma Community:

- Plugin page: <https://www.figma.com/community/plugin/843461159747178978/tokens-studio-for-figma>
- Or in Figma desktop: **Plugins → Find more plugins → search "Tokens Studio"**.

The plugin works in any Figma file. Open it via **Plugins → Tokens Studio for Figma** once installed.

## 2. Get a GitHub personal access token

The plugin pushes to git through your account.

1. Go to <https://github.com/settings/tokens?type=beta> → **Generate new token**.
2. Repository access: **Only select repositories** → pick the Nebutra-Sailor repo.
3. Permissions: **Contents = Read and write**, **Pull requests = Read and write**.
4. Copy the token — you will paste it into the plugin (not into git).

## 3. Configure the sync provider

In the plugin: **Settings → Sync providers → Add new → GitHub**.

Use these values exactly:

| Field                     | Value                                                  |
| ------------------------- | ------------------------------------------------------ |
| Name                      | `Nebutra Sailor`                                       |
| Personal access token     | _(paste from step 2)_                                  |
| Repository                | `Nebutra/Nebutra-Sailor` (replace if your fork differs)|
| Branch                    | `main`                                                 |
| File path                 | `packages/design-tokens/tokens`                        |
| Multi-file                | **On**                                                 |
| Themes path               | `.tokens-studio/themes.json`                           |
| Metadata path             | `.tokens-studio/metadata.json`                         |

Click **Save**.

## 4. First pull

1. In the plugin sidebar, click the cloud icon → **Pull from GitHub**.
2. The plugin loads `core`, `semantic`, and 8 theme sets. The token-set order matches `.tokens-studio/metadata.json`.
3. Switch between themes using the **Themes** dropdown — verify the 8 entries:
   - Base / Light, Base / Dark
   - Preset / Neon, Gradient, Dark Dense, Minimal, Vibrant, Ocean
4. If your selection is empty, the design-tokens package has not produced its first `tokens/**.json` yet. Pull again once a sibling PR lands.

## 5. Make a change & push back

1. Edit a token in the plugin (e.g. change `core.color.brand.primary`).
2. Click **Push to GitHub** → branch name `tokens-studio/<your-name>-<short-desc>`.
3. The plugin opens (or links to) a pull request.
4. CI (`design-sync.yml`) validates the diff and adds a sticky comment.
5. Wait for review + merge.

## Troubleshooting

| Symptom                           | Fix                                                                         |
| --------------------------------- | --------------------------------------------------------------------------- |
| "Could not load themes.json"      | Check the **Themes path** field — must be `.tokens-studio/themes.json`.    |
| "Branch already exists"           | Delete the stale local branch in the plugin or pick a new branch name.     |
| "Pull is empty"                   | The DTCG token files have not been generated yet — retry after the next merge to `main`. |
| "Push rejected: 403"              | Personal access token is missing **Contents: write** scope. Recreate it.   |
| "Theme matrix shows duplicates"   | Two entries share the same `id` in `themes.json` — open a bug.             |

---

## What designers should NOT do

- Do not edit `.tokens-studio/*.json` directly in Figma — these files describe the sync; the plugin updates them when needed.
- Do not check in changes to `packages/tokens/styles.css` from the plugin — that file is generated by CI from the DTCG source.
- Do not push to `main` directly — always go through a PR so the parity check runs.
